01

Ausgangslage & Leitidee

Der Optimizer ist heute ein Single-Purpose-Tool (Schiffs-Loadout). Es gibt noch keine Reiter-Navigation — alles liegt auf einer Seite. Die beiden neuen Bereiche machen daraus einen kompletten SC-Karriere-Begleiter: „Was soll ich spielen, um mein Ziel zu erreichen?" (Earnings Finder) → „Womit fliege ich das optimal?" (Loadout-Optimizer) → „Wie spiele ich das neue Feature?" (Patch Hub).

Leitidee: Der Earnings Finder beantwortet „welche Aktivität", der bestehende Optimizer „welches Schiff/Loadout", der Patch Hub „wie genau spielt man es". Drei Reiter, ein durchgängiger Flow — mit Querverlinkung zwischen ihnen.

Kern-Realität (prägt das ganze Konzept): Es gibt keine offizielle CIG-API für „aUEC pro Stunde". Diese Werte sind community-ermittelt (Guides wie ggwtb, mmopixel, RSI Community Hub, SC-Wiki, YouTube) und ändern sich mit jedem Patch / Wipe. Das Tool muss diese Daten daher kuratiert, patch-versioniert und mit Konfidenz-Kennzeichnung führen — nicht so tun, als wären sie exakt. Das ist der wichtigste konzeptionelle Punkt (siehe §6 Datenpflege).

02

Architektur-Integration

Der Optimizer bleibt rein statisch (ES-Module, kein Build, Start via python http.server, GitHub Pages). Die neuen Reiter fügen sich nahtlos ein:

  • Top-Level-Reiter-Navigation einführen (im app-header): [ Loadout Optimizer ] [ Earnings Finder ] [ Patch Hub ]. Der gesamte heutige Inhalt wird zum ersten Reiter — null Funktionsverlust. Tab-State per Hash-Route (#optimizer / #earnings / #patch) + localStorage „last tab".
  • Neue ES-Module analog zum Bestand: js/earnings-finder.js, js/patch-hub.js, geladen über denselben data-loader.js (fetch + cache).
  • Neue Daten-Files in data/: activities.json, guides.json, factions.json — gleiches { "data": [...] }-Pattern wie ships.json.
  • Design-System wiederverwenden: dieselben CSS-Variablen (--sc-teal, Terminal/CRT-Look, .panel, .ship-list-Muster) → konsistente Optik.
  • Querverlinkung: Eine Activity-Karte hat „⟶ Optimize a ship for this" (springt in den Loadout-Reiter mit vorausgewähltem Schiff/Profil) und „⟶ How to play" (springt in den passenden Patch-Hub-Guide).

Oben stellt man sein Ziel + Rahmenbedingungen ein; darunter erscheint ein gerankte Liste der besten Aktivitäten mit transparenter Begründung.

D · Scoring-Logik (transparent)

Jede Aktivität wird pro Modus normalisiert & gewichtet gescort — die Faktoren sind im UI einsehbar (kein „Black Box"-Ranking):

score = wMoney · norm(auecPerHour)
      + wRep   · norm(repPerHour[selectedFaction])
      + wFun   · interestScore
      − riskPenalty · (risk · (1 − riskTolerance))
      − gateLockout            // rep/ship nicht erfüllt → ans Ende + „locked"-Badge
      × confidence             // estimated < verified
// Modus setzt die Gewichte: MaxaUEC → wMoney=1; Reputation → wRep=1;
// Mix → Slider; Fun → wFun=1

Gegatete Aktivitäten (z. B. Adagio braucht Senior Contractor) werden nicht versteckt, sondern als locked markiert mit Hinweis „so schaltest du es frei" → Brücke zum Reputation-Modus.

04

Reiter 2 — Patch Hub & Guides

Zeigt pro Patch, was neu/geändert ist, und liefert Step-by-Step-Guides für komplexe Features. Beantwortet „Was kann ich seit dem letzten Patch Neues spielen — und wie genau?".

Aufbau

  • Patch-Timeline oben (4.8 · 4.8.1 · …) + Feature-Karten mit NEW/CHANGED-Badge, Kategorie-Filter (Gameplay-Loop · Event · Mechanik · Schiff).
  • Guide-Detail (Klick auf Feature): nummerierte Schritte, Voraussetzungen, Belohnung, Tipps, Schwierigkeit, Zeitaufwand, externe Links (RSI Community Hub / Wiki / Video).

Paradebeispiel-Guide: Hathor Orbital Laser Platform

Kategorie: Event/Loop · Schwierigkeit: ●●●○ · ~45–90 min · Crew 2–4 empfohlen
  1. Mission annehmen beim Mission-Giver (Wicko).
  2. Boden-Satelliten ausrichten auf den Monden Dayar & Abedine.
  3. Comp-Boards einsammeln und zur Plattform bringen.
  4. Orbital-Laser mit Energie versorgen / aktivieren (Align-and-Fire).
  5. Carite abbauen aus dem gespaltenen Felsen.
  6. Belohnung abholen → koppelt zurück an den Earnings Finder (aUEC + Rep).

Quelle/Verweis pro Guide hinterlegt (RSI Community Hub „Hathor Laser One-Page Guide", Wiki). So bleibt jeder Guide nachprüfbar.

Weitere 4.8-Kandidaten für Guides: Tactical Strike Groups (Multi-Ship-Coop), Refueling-System (Starlite), G-Force/Flight-Suits, Ship-Combat V3, Post-Wipe-Progression.

06

Datenpflege — der kritische Teil

Ohne Pflege veralten die aUEC/h-Zahlen mit dem nächsten Patch sofort. Drei Bausteine:

  • Patch-Versionierung & Konfidenz: jede Zahl trägt patch + confidence (verified/estimated/stale). UI zeigt „Stand: Patch 4.8.1 · est." + Community-Disclaimer. Stale-Einträge werden optisch gedämpft.
  • Kuratierte Quelle: Werte aus Community-Guides (ggwtb, mmopixel, RSI Community Hub, SC-Wiki, YouTube) — manuell verifiziert, nie als exakt verkauft (Ranges + „est.").
  • Optionale KI-Update-Pipeline v2: ein Recherche-Agent (analog deiner BIT-Portal-Maintenance-Pipeline) zieht pro Patch die aktuellen Methoden, schlägt activities.json-Deltas als PR vor — du bestätigst. Hält das Tool ohne manuellen Dauer-Aufwand aktuell.
Disclaimer im UI: „Community-ermittelte Richtwerte, kein offizieller CIG-Wert — variiert je Skill, Server, Patch." Schützt Glaubwürdigkeit.

  • activities.json / guides.json / factions.json + Konfidenz/Patch-Stand-Anzeige

  • Reputation-Modus mit Fraktions-Trade-offs (rauf bei A = runter bei B) + Gate-Aufschluss-Pfade

Offene Entscheidungen

1 · Daten-Tiefe zum Start: Wie viele Aktivitäten + Guides im MVP (z. B. Top 20 Aktivitäten / 5 Guides) — und kurate ich sie autonom aus den Community-Quellen, oder lieferst du eine Prioliste?
2 · Fun/„interessant" definieren: Welche Tags zählen als „cool" (Events, Story-Missionen, Scenic, Easter-Eggs)? Gibt es konkrete Beispiele, die rein sollen?
3 · KI-Update-Pipeline: direkt mit einplanen (v3) oder erst manuell pflegen und später automatisieren?
4 · Scope MVP: beide Reiter zusammen, oder erst der Earnings Finder, dann der Patch Hub?
5 · Sprache: wie beim Optimizer komplett Englisch (für die Community), richtig?
